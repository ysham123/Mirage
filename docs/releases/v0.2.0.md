# Mirage v0.2.0

Mirage v0.2.0 ships the production runtime gateway, the first framework
integration, and a reproducible benchmark harness. This is the release
that turns Mirage from an interesting OSS project into a production
runtime policy gateway with measurable numbers.

The mission sentence is the contract: same YAML policy file, gateway
in production and gate in CI, no LLM in the decision loop. v0.2.0 is
the first version where every word of that sentence is backed by code,
docs, examples, and tests on the main branch.

## What ships

The runtime gateway is now first-class. `mirage.gateway` and the
`mirage gateway` CLI evaluate the same `policies.yaml` against real
upstream traffic in either passthrough (forward + log) or enforce
(forward when policy passes, block with HTTP 403 when it fails) mode.
Trace events carry a `mode` discriminator and a unified outcome
taxonomy (`allowed` / `flagged` / `blocked` / `error`).

The policy DSL gains eleven new safety-relevant operators
(`regex_match`, `not_regex_match`, `contains`, `not_contains`,
`starts_with`, `not_starts_with`, `ends_with`, `length_lte`,
`length_gte`, `host_in`, `host_not_in`). Five real-world example
policies under `examples/policies/` cover the production risks an
agent deployment actually creates today: PII leaks, prompt-injection
markers, outbound URL allowlists, monetary cost caps, and output
length caps.

Containment rate, decision latency, and time-to-decide are now
captured by the runtime and surfaced in the console. The new
`mirage.metrics.ContainmentMetrics` dataclass formalizes the formula:
`blocked / max(1, blocked + policy_violation + flagged)`. The
`/api/runs/{run_id}/containment` endpoint exposes per-run metrics; the
existing `/api/metrics/overview` exposes a fleet-wide rollup. The
Next.js console gets a single Containment row in the run-detail stats
bar.

Mirage now has a real framework integration. `mirage.integrations.\
openai_agents.wrap_with_mirage` returns a copy of an OpenAI Agents SDK
`Agent` whose tool calls are policy-checked through a Mirage gateway
before the underlying tool runs. Blocked tool calls surface as
`MiragePolicyBlockedError`; allowed tool calls fall through. Install
with `pip install mirage-ci[openai-agents]`; documented in
`docs/INTEGRATIONS_OPENAI_AGENTS_SDK.md`.

The benchmark harness under `benchmarks/` is reproducible. `make
bench` runs three synthetic scenarios (PII leak, prompt injection,
cost runaway) against a Mirage gateway in enforce mode and reports
containment rate, false-positive rate, precision, and latency
percentiles. `BENCHMARKS.md` documents methodology and is honest about
limitations (synthetic data, single-process, file-based policy reload
on every request).

## How to upgrade

```bash
pip install --upgrade mirage-ci
```

The public Python API (`MirageSession`, `MirageEngine`, `MirageResult`,
`MirageGateway`, `GatewayResult`, `PolicyDecision`, `PolicyEvaluator`)
is unchanged. CI mode behavior is unchanged: the procurement harness,
the four CI-mode outcomes, and `gate-run` exit codes work byte-for-
byte the same as v0.1.3.

If you were depending on `0.2.0.dev0` from a checkout, the published
v0.2.0 is the same surface plus the new operators, integrations, and
metrics described above.
